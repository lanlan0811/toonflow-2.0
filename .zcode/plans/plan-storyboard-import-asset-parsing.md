# 分镜表资产统计与关联修复计划

## 目标

修复“基于分镜表”项目中的三个确定性问题，并保证解析预览、资产统计、提交入库和分镜关联使用同一份有效资产结果：

1. 角色参考有 4 项、实际分镜只使用 3 项时，主统计显示角色 3，而不是 4。
2. 场景参考有 3 项、分镜明细另有“片尾画面”时，主统计显示场景 4，并正常创建第四个场景。
3. 同批次已明确存在“公园长椅”道具时，场景或画面描述“小区公园长椅旁”所在镜头自动关联“小区公园（scene）”和“公园长椅（tool）”。

本次不使用模糊 AI 推断，也不把“工作人员”静默归并为“社区工作人员”；未使用的角色参考只给出明确警告，避免违背当前“实际角色为 3”的业务判断。

## 阶段 1：统一资产语义与统计

新增无 Express、无数据库副作用的共享模块 `src/lib/storyboardImportAssets.ts`：

1. 统一名称清理和按类型去重。
2. 从最终分镜行的 `roleNames/sceneNames/toolNames` 计算有效资产集合及统计。
3. 对整批分镜执行第二遍关联补全：
   - 第一遍从所有行已有 `toolNames` 收集同批次明确道具词典；
   - 第二遍用完整道具名扫描每行的原始场景、画面内容和视频描述；
   - 命中后补入该行 `toolNames`；
   - 不从普通文本凭空创建道具，不做语义模糊匹配。
4. 识别未被分镜使用的角色参考，以及缺少场景美术参考的行级场景。
5. 提供提交端统计一致性比较，防止解析展示与提交数据漂移。

## 阶段 2：修复解析结果

修改 `src/routes/storyboardImport/parse.ts`：

1. 保留现有 TXT、Markdown、DOCX、JSON、CSV 第一遍解析逻辑。
2. 所有格式统一经过批次级第二遍道具关联补全。
3. 以补全后的行级有效资产生成 `meta.assetStats`。
4. 增加聚合 warning：
   - 角色参考未被任何分镜使用时，提示未计入待入库资产；
   - 行级场景不在场景美术参考中时，提示将按场景名称入库。
5. 导出可测试的解析入口，不改变默认 Express 路由。

## 阶段 3：修复提交一致性

修改 `src/routes/storyboardImport/commit.ts`：

1. `ImportMeta` 和请求校验正式接收可选 `assetStats`。
2. 提交端再次执行批次级关联补全，避免直接调用提交接口绕过解析规则。
3. 携带 `assetStats` 时，与服务端重算结果比较；不一致则拒绝提交并要求重新解析。旧客户端未携带统计时保持兼容。
4. 使用补全后的分镜行创建资产及写入 `o_assets2Storyboard`。
5. 命名资产创建或查询后若无法取得 ID，则抛错回滚，不再静默漏关联。
6. 自动补入的道具不使用当前镜头无关的 `props` 覆盖正确资产描述；只有当前 `props` 明确列出该道具时才写入上下文。

## 阶段 4：前端预览加固

修改 `data/web/secondary-dev-patch.js`：

1. 无后端统计时的兼容统计只使用行级资产，不再混入未使用参考项。
2. 解析预览同时展示原始场景与识别场景资产。
3. 保持现有提交、列表和手工编辑关联行为。

## 阶段 5：自动化回归测试

新增：

- `test/storyboardImport/assets.test.ts`
- `test/storyboardImport/parse.test.ts`

并在 `package.json` 增加 `test:storyboard-import`，使用现有 `tsx` 和 Node 内置 `node:test`，不增加测试框架依赖。

重点覆盖：

1. 角色参考 4、有效角色 3。
2. 场景参考 3、有效场景 4。
3. 较早镜头通过同批次道具词典补关联“公园长椅”。
4. 不存在明确道具词条时不凭空创建。
5. 自动补关联道具不采用不相关 `props` 描述。
6. JSON/CSV 只有行级资产时统计正确。
7. Markdown/TXT 均经过统一后处理。
8. 旧统计与服务端有效统计不一致时能够被识别。

测试不得导入真实数据库初始化链，也不得修改 `data/db2.sqlite`。

## 阶段 6：验证和生产构建

依次执行：

1. `yarn test:storyboard-import`
2. `yarn lint`
3. `node --check data/web/secondary-dev-patch.js`
4. `yarn build`
5. `node --check data/serve/app.js`
6. `git diff --check`
7. 检查最终 Git 状态，确认没有修改真实数据库或无关文件。

生产后端包 `data/serve/app.js` 必须由现有构建流程生成，不手工编辑。

## 预期结果

以当前问题样例为准：

- 解析统计显示角色 3、场景 4；
- “社区工作人员”显示为未使用参考警告，不创建第四个角色；
- “片尾画面”显示为缺少场景美术参考警告，但作为第四个场景入库；
- 第一镜关联资产包含“小区公园 · 场景”和“公园长椅 · 道具”；
- 解析预览、提交后的资产管理区和分镜列表关联资产保持一致。
