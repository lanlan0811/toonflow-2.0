## 1. 目标流程设计

建议把三种入口统一收敛成同一个“短剧生产流水线”，区别只在前置输入阶段。

### 1.1 三种项目类型

#### A. 基于小说原文

推荐流程：

1. 新建项目：`projectType = novel`
2. 导入小说章节：调用 `/api/novel/addNovel`
3. 提取章节事件：复用小说事件提取逻辑
4. 使用 ScriptAgent 生成/整理剧本
5. 进入统一生产链路：
   - 提取原始资产
   - 生成原始资产图
   - 生成衍生资产
   - 生成衍生资产图
   - 制作分镜面板
   - 生成分镜图
   - 生成视频提示词
   - 生成视频

#### B. 基于剧本

推荐流程：

1. 新建项目：`projectType = script`
2. 单集/批量导入剧本：调用 `/api/script/addScript` 或 `/api/script/batchAddScript`
3. 调用 `/api/script/extractAssets` 提取角色、场景、道具
4. 进入统一生产链路：资产图 → 衍生资产 → 分镜面板 → 分镜图 → 视频

#### C. 基于分镜表

推荐流程：

1. 新建项目：`projectType = storyboard`
2. 上传/粘贴/导入分镜表
3. 将分镜表解析成结构化数据：
   - `prompt`
   - `duration`
   - `track`
   - `videoDesc`
   - `shouldGenerateImage`
   - `associateAssetsIds`
4. 调用 `/api/production/storyboard/batchAddStoryboardInfo` 写入分镜面板
5. 如果分镜表中包含角色/场景/道具列，则新增或匹配原始资产
6. 进入统一生产链路：资产图 → 衍生资产 → 分镜图 → 视频

## 2. 后端改造计划

### 阶段 1：统一项目类型与流程状态

1. 增加一个后端常量/类型文件，例如：
   - `src/constants/project.ts`
   - 定义：
     - `PROJECT_TYPE_NOVEL = "novel"`
     - `PROJECT_TYPE_SCRIPT = "script"`
     - `PROJECT_TYPE_STORYBOARD = "storyboard"`

2. 对 `/api/project/addProject` 增加项目类型枚举校验。  
   现状是 `projectType: z.string()`，见 `D:\Trae项目\Toonflow-app-master\src\routes\project\addProject.ts:10`。  
   建议改为 `z.enum(["novel", "script", "storyboard"])` 或兼容中文值映射。

3. 增加一个“流程状态聚合接口”，例如：
   - `POST /api/production/workflow/getProgress`

   返回当前项目/剧本的阶段状态：
   - 小说导入状态
   - 剧本导入状态
   - 原始资产数量/生成状态
   - 衍生资产数量/生成状态
   - 分镜数量/分镜图状态
   - 视频轨道数量/视频生成状态

   这样前端右上角流程按钮可以根据这个接口判断哪些按钮可点、哪些已完成、哪些需要轮询。

### 阶段 2：补齐“基于分镜表”的后端能力

目前已有结构化分镜批量写入接口，但缺少“分镜表文件解析”接口。建议新增：

1. `POST /api/storyboardImport/parse`
   - 输入：Excel/CSV/JSON/纯文本内容，或 base64 文件
   - 输出：标准化分镜数组，不直接入库
   - 作用：让用户先预览解析结果

2. `POST /api/storyboardImport/commit`
   - 输入：用户确认后的标准化分镜数组
   - 内部调用或复用 `batchAddStoryboardInfo` 的落库逻辑
   - 自动创建/匹配资产
   - 自动创建 `o_videoTrack`

3. 分镜表推荐标准字段：
   - 集数/场次
   - 镜头序号
   - 画面描述
   - 分镜图提示词
   - 视频描述
   - 时长
   - 角色
   - 场景
   - 道具
   - 是否生成分镜图
   - 分组/轨道名

### 阶段 3：统一资产生成链路

1. 明确资产层级：
   - 原始资产：`o_assets.assetsId IS NULL`
   - 衍生资产：`o_assets.assetsId = 父资产 ID`

2. 新增/整理一组流程接口：
   - `POST /api/production/workflow/extractOriginalAssets`
   - `POST /api/production/workflow/generateOriginalAssetImages`
   - `POST /api/production/workflow/generateDerivedAssets`
   - `POST /api/production/workflow/generateDerivedAssetImages`

3. 底层尽量复用现有接口：
   - 剧本资产提取复用 `/api/script/extractAssets`
   - 资产图生成复用 `/api/assetsGenerate/batchGenerateImageAssets`
   - 衍生资产生成优先复用 ProductionAgent 工具或 `src/routes/production/assets/**`

4. 对前端展示做统一：
   - 资产卡片展示：名称、类型、描述、图片状态、错误原因
   - 衍生资产展示：绑定父资产，展示生成来源

### 阶段 4：统一分镜面板与分镜图生成

1. 分镜面板写入继续复用 `/api/production/storyboard/batchAddStoryboardInfo`。
   - 该接口已经会按 `track` 自动创建/复用 `o_videoTrack`，核心逻辑见 `D:\Trae项目\Toonflow-app-master\src\routes\production\storyboard\batchAddStoryboardInfo.ts:60`。

2. 分镜图生成继续复用 `/api/production/storyboard/batchGenerateImage`。
   - 该接口会读取关联资产图作为参考图，见 `D:\Trae项目\Toonflow-app-master\src\routes\production\storyboard\batchGenerateImage.ts:51`。

3. 建议补齐：
   - 分镜图批量重试
   - 只生成失败项
   - 只生成未生成项
   - 强制重生成
   - 生成进度汇总接口

### 阶段 5：统一视频工作台

1. 视频提示词生成：复用 `/api/production/workbench/batchGeneratePrompt`
2. 视频生成：复用 `/api/production/workbench/batchGenerateVideo`
3. 视频状态轮询：复用 `/api/production/workbench/checkVideoStateList`
4. 视频选择：复用 `/api/production/workbench/selectVideo`

同时需要修复一个状态不统一问题：

- `batchGenerateVideo.ts` 成功写入的是 `生成成功`，见 `D:\Trae项目\Toonflow-app-master\src\routes\production\workbench\batchGenerateVideo.ts:125`。
- 部分读取展示逻辑更偏向 `已完成`。

建议统一成一种后端状态，或在 API 层兼容：

```ts
生成中 -> generating
生成成功 / 已完成 -> success
生成失败 -> failed
未生成 -> idle
```

## 4. 必须优先修复/规避的问题

1. **前端源码缺失**  
   - 需要确认 Toonflow-web 源码位置。
   - 不建议直接修改 `data/web/index.html`，这是打包产物。

2. **视频状态命名不统一**  
   - 建议兼容 `生成成功` 和 `已完成`，避免生成成功后前端显示成未生成。

3. **剧本资产轮询状态疑似不正确**  
   - `extractState` 实际是数字状态 `2/0/1/-1`。
   - 轮询接口需要按数字状态判断，不应按字符串 `生成中` 判断。

4. **批量异步任务缺少 batchId**  
   - 当前很多批量接口是“立即返回 + 后台执行 + 前端轮询业务对象”。
   - 为了完整流程体验，建议增加 `batchId` 或统一任务聚合状态。

5. **ID 生成方式存在并发风险**  
   - 多处使用 `Date.now()` 作为 ID。
   - 新增接口尽量用 SQLite 自增 ID 或 `uuid`。

6. **分镜表文件解析能力缺失**  
   - “基于分镜表”真正落地的关键是新增解析/字段映射/预览确认流程。

## 5. 推荐实施顺序

### 第一期：最小可用闭环

目标：优先跑通“基于分镜表 → 资产 → 分镜图 → 视频”的完整闭环。

1. 后端新增项目类型常量与校验
2. 后端新增分镜表解析/提交接口
3. 前端新建项目支持三种项目类型
4. 前端实现分镜表导入、预览、提交
5. 前端复用现有分镜面板与分镜图生成接口
6. 前端复用现有视频工作台接口
7. 修复视频状态兼容问题

### 第二期：统一三种入口流程

目标：小说、剧本、分镜表都进入同一个生产流程。

1. 新增流程进度聚合接口
2. 前端实现统一流程导航条
3. 剧本入口接入资产提取
4. 小说入口接入事件提取和剧本生成
5. 所有入口最终统一进入资产/分镜/视频工作台

### 第三期：体验与稳定性增强

目标：提升批量生成稳定性和可维护性。

1. 增加 batchId 任务聚合
2. 统一生成状态枚举
3. 增加失败重试、取消、只重试失败项
4. 增加 mock vendor 方便无真实模型时调试
5. 梳理 `o_videoTrack.videoId/selectVideoId` 字段语义
6. 增加必要的类型定义和接口测试
