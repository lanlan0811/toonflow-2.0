## 二次开发分支计划：基于分镜表入口重构 V2

### 一、目标修正

这次二次开发分支的核心目标是把“基于分镜表”从“剧本导入/添加剧本”的思路中拆出来，做成独立入口与独立页面：

1. 新建项目选择 **“基于分镜表”** 后，项目卡片标签必须显示 **“基于分镜表”**。
2. “基于分镜表”项目不再进入 `/script` 剧本管理页，而是进入新页面 **“分镜表管理”**。
3. 分镜表管理页参考“剧本管理”的页面结构，但功能专属于分镜表：
   - 搜索分镜表/镜头
   - 新建分镜表
   - 支持上传/填写 TXT、Word DOCX、Markdown 分镜表
   - 解析预览
   - 确认导入
   - 在本页面生成/管理基于分镜表提取出的原始资产
4. 分镜表格式以用户提供的三个文件为标准：
   - `短剧分镜表/分镜表_标准格式.txt`
   - `短剧分镜表/分镜表_标准表格版.docx`
   - `短剧分镜表/分镜表.md`
5. 后续流程调整为：
   - 填写/导入分镜表
   - 基于分镜表生成原始资产
   - 生成原始资产图片
   - 生成衍生资产
   - 生成衍生资产图片
   - 制作分镜面板（基于分镜表、原始资产、衍生资产）
   - 生成分镜图
   - 生成视频

---

### 二、当前代码现状

#### 1. 项目类型后端已支持 `storyboard`

已确认：

- `src/constants/project.ts` 已有：
  - `novel` -> `基于小说原文`
  - `script` -> `基于剧本`
  - `storyboard` -> `基于分镜表`
- `/api/project/addProject` 已能规范化 `storyboard / 分镜表 / 基于分镜表`。
- 数据库 `o_project.projectType` 已有承载能力。

所以后端项目类型本身不需要大改，重点是前端展示和跳转逻辑。

#### 2. 项目卡片标签显示存在二分逻辑问题

当前前端打包产物中，项目卡片标签逻辑仍类似：

```js
projectType === "novel" ? "基于小说原文" : "基于小说剧本/基于剧本"
```

这会导致所有非 `novel` 的项目，包括 `storyboard`，都被显示成剧本类标签。

需要改成三值映射：

```ts
novel      -> 基于小说原文
script     -> 基于剧本
storyboard -> 基于分镜表
```

#### 3. 当前没有正式的“分镜表管理”页面

当前仓库没有可维护的前端源码，主要前端是：

- `data/web/index.html`：打包后的 Vue 前端产物
- `data/web/secondary-dev-patch.js`：二开浮层补丁
- `data/web/secondary-dev-patch.css`：二开样式补丁

现有二开能力在 `secondary-dev-patch.js` 中以右下角“二开流程”浮层形式存在，并不是真正的路由页面。

正式实现建议优先在原前端源码仓库中完成；如果暂时拿不到前端源码，则继续使用 `secondary-dev-patch.js/css` 做注入式页面，但维护性会低于源码实现。

#### 4. 分镜表导入后端已有基础接口，但解析能力不足

已有接口：

- `POST /api/storyboardImport/parse`
- `POST /api/storyboardImport/commit`

现状：

- `parse.ts` 支持 JSON / CSV / TSV / 简单纯文本。
- 不支持用户提供的 TXT 标准键值块格式。
- 不支持 Markdown 管道表格格式。
- 不支持 DOCX 表格解析。
- 时长如 `4秒`、`4s` 当前不能稳定归一化。
- `镜号`、`景别`、`镜头运动`、`台词/旁白`、`音效/配乐`、`道具/陈设`、`备注` 等标准字段没有完整建模。

#### 5. `commit.ts` 已能自动创建资产，但资产质量需要增强

当前 `commit.ts` 会根据：

- `roleNames`
- `sceneNames`
- `toolNames`

自动创建：

- 角色资产
- 场景资产
- 道具资产

但不足是：

- 资产描述只是名称本身。
- 没有利用“角色形象参考”和“场景美术参考”。
- 没有写入 `o_storyboard.index`。
- 没有事务包裹。
- 内部仍会创建“分镜表导入”占位剧本，用于兼容现有数据模型。

本阶段可以保留内部占位剧本机制，但前端页面不再把它表现为“添加剧本”。

### 阶段 2：增强分镜表解析接口

文件：

- `src/routes/storyboardImport/parse.ts`

#### 2.1 扩展请求格式

建议兼容现有请求，并增加：

```ts
{
  content?: string;
  base64?: string;
  filename?: string;
  mimeType?: string;
  format?: "auto" | "json" | "csv" | "text" | "txt-standard" | "markdown" | "docx";
}
```

#### 2.2 支持 TXT 标准格式

参考：

- `短剧分镜表/分镜表_标准格式.txt`

需要解析：

- 项目信息：制作规格、总时长、主题
- 分镜明细：
  - 镜号
  - 时长
  - 景别
  - 镜头运动
  - 场景
  - 画面内容
  - 台词/旁白
  - 音效/配乐
  - 道具/陈设
  - 备注
- 角色形象参考
- 场景美术参考
- 音乐音效总览
- 字幕制作说明

解析结果中的每条分镜统一转换为：

```ts
{
  shotNo: string;
  prompt: string;
  duration: number;
  track: string;
  videoDesc: string;
  shouldGenerateImage: number;
  roleNames: string[];
  sceneNames: string[];
  toolNames: string[];
  shotSize?: string;
  cameraMove?: string;
  visualContent?: string;
  dialogue?: string;
  audio?: string;
  props?: string;
  remark?: string;
}
```

#### 2.3 支持 Markdown 分镜表

参考：

- `短剧分镜表/分镜表.md`

需要解析 Markdown 管道表格：

```md
| 镜号 | 时长 | 景别 | 镜头运动 | 场景 | 画面内容 | 台词/旁白 | 音效/配乐 | 道具/陈设 | 备注 |
```

处理规则：

- `<br>`、`<br><br>` 转换为换行。
- 去除或合理保留 Markdown 加粗标记。
- `4s`、`4秒` 等统一转为数字秒数。
- `场景` 写入 `sceneNames`。
- `道具/陈设` 拆分写入 `toolNames`。
- 角色从“角色形象参考”表和画面内容中匹配。

#### 2.4 支持 Word DOCX 表格版

参考：

- `短剧分镜表/分镜表_标准表格版.docx`

建议新增依赖：

- `mammoth`：读取 DOCX 并保留表格 HTML
- `cheerio`：解析 mammoth 输出的 HTML 表格

解析目标表格：

1. 项目信息表
2. 分镜明细表
3. 角色形象参考表
4. 场景美术参考表
5. 音乐音效总览表
6. 台词语速核对表
7. 字幕制作说明表

#### 2.5 返回结构增加 `meta` 和 `warnings`

建议返回：

```ts
{
  data: ImportRow[];
  meta: {
    project?: {
      productionSpec?: string;
      totalDuration?: number;
      theme?: string;
    };
    roles?: RoleSpec[];
    scenes?: SceneSpec[];
    music?: MusicSpec[];
    subtitle?: SubtitleSpec;
  };
  total: number;
  warnings: string[];
}
```

这样前端“分镜表管理”页面可以展示：

- 分镜预览
- 角色资产预览
- 场景资产预览
- 解析警告

### 阶段 3：增强分镜表提交接口

文件：

- `src/routes/storyboardImport/commit.ts`

#### 3.1 扩展入参

兼容现有字段，并增加：

```ts
{
  projectId: number;
  scriptId?: number | null;
  scriptName?: string | null;
  data: ImportRow[];
  meta?: {
    roles?: RoleSpec[];
    scenes?: SceneSpec[];
  };
  options?: {
    createScriptAssets?: boolean;
    useReferenceAssetDescriptions?: boolean;
    writeStoryboardIndex?: boolean;
  };
}
```

#### 3.2 写入分镜顺序

- 使用导入顺序写入 `o_storyboard.index`。
- `shotNo` 如 `3a`、`3b` 保留在 `videoDesc` 中。
- 若数据库后续允许，可考虑新增 `shotNo` 字段；本阶段为了减少数据库迁移风险，先不强制新增字段。

#### 3.3 提升自动创建资产质量

角色资产描述使用“角色形象参考”：

```txt
角色：林小雨
年龄：...
外貌特征：...
服装：...
性格关键词：...
```

场景资产描述使用“场景美术参考”：

```txt
场景：小区公园
时间：...
色调：...
元素：...
氛围：...
```

道具资产描述使用：

```txt
道具名 + 出现镜头 + 道具/陈设上下文
```

#### 3.4 使用事务

`commit` 应用事务包裹：

- 创建内部占位剧本
- 创建/更新资产
- 插入分镜
- 写入资产与分镜关联
- 创建/更新视频轨道

避免导入失败时只写入一半数据。

#### 3.5 关于内部占位剧本

当前数据模型中分镜、视频轨道等仍依赖 `scriptId`。

本阶段建议：

- 后端继续内部创建占位剧本以兼容现有链路。
- 前端不展示“添加剧本”概念。
- 占位剧本可命名为：`分镜表导入-YYYYMMDD-HHmm`。
- 后续如果要彻底去掉占位剧本，再单独设计数据库结构调整。

## 六、流程按钮与业务流程调整

“基于分镜表”项目的推荐流程：

1. 分镜表管理
   - 上传/填写分镜表
   - 解析预览
   - 确认导入
2. 原始资产管理
   - 从分镜表生成原始资产
   - 编辑/确认角色、场景、道具
3. 生成原始资产图片
4. 生成衍生资产
5. 生成衍生资产图片
6. 制作分镜面板
   - 数据来源：分镜表 + 原始资产 + 衍生资产
7. 生成分镜图
8. 生成视频提示词
9. 生成视频

对于 `storyboard` 项目，流程中的“提取原始资产”不应表现为“从剧本中提取”，而应表现为：

```txt
从分镜表生成原始资产
```

因此工作流配置可以做项目类型文案适配：

- `script`：提取原始资产
- `storyboard`：从分镜表生成原始资产
- `novel`：从小说/剧本链路提取原始资产

## 七、验收标准

### 1. 新建项目与项目标签

- 新建项目时选择“基于分镜表”。
- 项目卡片右上角标签显示 **“基于分镜表”**。
- 不显示“基于小说剧本”。
- 不显示“基于小说原文”。

### 2. 路由和页面

- 点击“基于分镜表”项目，进入 **分镜表管理** 页面。
- 不进入“剧本管理”页面。
- 页面菜单/顶部图标为分镜表专属图标。
- 页面标题为 **“分镜表管理”**。

### 3. 分镜表创建

- 页面只有 `+ 新建分镜表`。
- 不出现 `+ 新建剧本`。
- 不出现 `+ 批量上传剧本`。

### 4. 文件格式支持

以下文件均可解析并预览：

- `短剧分镜表/分镜表_标准格式.txt`
- `短剧分镜表/分镜表_标准表格版.docx`
- `短剧分镜表/分镜表.md`

解析后至少能正确得到：

- 镜号
- 时长
- 景别
- 镜头运动
- 场景
- 画面内容
- 台词/旁白
- 音效/配乐
- 道具/陈设
- 备注

### 5. 原始资产生成

在分镜表管理页内能看到并处理：

- 角色资产
- 场景资产
- 道具资产

资产描述应优先来自：

- 角色形象参考
- 场景美术参考
- 分镜明细中的道具/陈设

### 6. 后续生产链路

从分镜表管理页可以继续完成：

- 生成原始资产图片
- 生成衍生资产
- 生成衍生资产图片
- 制作分镜面板
- 生成分镜图
- 生成视频

### 7. 数据兼容

- 后端可继续内部创建占位剧本以兼容现有 `scriptId` 数据模型。
- 前端不把这个过程展示为“添加剧本”。
- 已有剧本项目、小说项目功能不受影响。

## 八、建议实施顺序

### 第一阶段：快速修正体验错误

1. 修复项目卡片标签：`storyboard -> 基于分镜表`。
2. 修复项目点击跳转：`storyboard -> 分镜表管理`。
3. 新增/注入“分镜表管理”入口与专属图标。
4. 移除分镜表流程在“添加剧本”中的呈现。

### 第二阶段：完成分镜表管理页

1. 新增“分镜表管理”页面布局。
2. 支持粘贴/上传 `.txt`、`.md`、`.docx`。
3. 接入 `/api/storyboardImport/parse`。
4. 接入 `/api/storyboardImport/commit`。
5. 展示分镜表列表和解析预览。

### 第三阶段：增强解析能力

1. 实现 TXT 标准格式解析。
2. 实现 Markdown 表格解析。
3. 新增 DOCX 表格解析依赖并实现解析。
4. 统一时长、镜号、场景、角色、道具等字段。
5. 增加解析警告。

### 第四阶段：原始资产生成内嵌到页面

1. 在分镜表管理页展示角色/场景/道具资产。
2. 从角色形象参考、场景美术参考生成资产描述。
3. 接入原始资产提示词润色。
4. 接入原始资产图片生成。
5. 支持失败重试。

### 第五阶段：完整生产链路串联

1. 接入生成衍生资产。
2. 接入生成衍生资产图片。
3. 接入制作分镜面板。
4. 接入生成分镜图。
5. 接入生成视频。
6. 完成流程状态轮询与异常提示。

## 九、实现方式选择

### 推荐方式：拿到原前端源码后实现

优点：

- 路由、菜单、组件、状态管理都更稳定。
- 可以真正复用“剧本管理”页面组件。
- 后续维护成本低。

### 当前仓库可执行的方式：二开补丁方式

如果暂时没有原前端源码，则在当前仓库中只能继续基于：

- `data/web/index.html`
- `data/web/secondary-dev-patch.js`
- `data/web/secondary-dev-patch.css`

做注入式页面。

优点：

- 可以在当前仓库直接落地。
- 可以快速修复用户截图中的错误。

缺点：

- 依赖打包产物结构。
- 不如源码开发稳定。
- 后续前端重新打包可能覆盖修改。

本计划建议：

1. 如果能提供前端源码，优先在源码中实现。
2. 如果不能提供源码，则先用二开补丁落地 `分镜表管理` 页面，确保业务闭环可用。
